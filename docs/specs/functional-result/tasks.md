# Implementation Plan

## 1. 基盤：契約・例外・ランタイム方針

- [ ] 1.1 (P) ログエントリの共通マーカー契約を定義する
  - ログ出力先・形式・永続化を要求しないマーカー契約として公開する
  - 契約単体で具象実装を強制しないことを確認できる
  - _Requirements: 4.5, 7.3, 7.4_
  - _Boundary: LogEntry_

- [ ] 1.2 パイプ演算子の検証方針を確定する
  - プロジェクトの PHP 制約を確認し、パイプ構文を実行するテストを含めるか、利用例を PHPDoc のみとするかを決定する
  - 選択結果が後続の合成検証タスクの前提として明確である
  - _Requirements: 3.5, 6.5_

- [ ] 1.3 (P) 失敗 Result の不正な値取り出しを示す専用例外を用意する
  - 失敗状態での成功値取得が不正であることを伝える例外型が利用可能である
  - _Requirements: 2.2_
  - _Boundary: ResultUnwrapException_

## 2. Result の生成・状態・ログ保持

- [ ] 2.1 成功・失敗の排他的な Result を生成・判定できるようにする
  - 成功値のみ、失敗理由のみを保持し、両方が同時に有効にならない
  - 成功・失敗の状態判定が生成直後から一貫して機能する
  - _Requirements: 1.1, 1.2, 1.3, 1.4, 1.5_
  - _Depends: 1.1_

- [ ] 2.2 ログ collection を付随させ、省略時は空 collection とする
  - 生成時にログを指定でき、省略時は空のログ collection が保持される
  - 保持したログ collection を参照でき、外部ログシステムへの出力責務は持たない
  - _Requirements: 4.1, 4.2, 4.3, 7.1, 7.2_

## 3. 値の取得と失敗理由の参照

- [ ] 3.1 成功時の値取得と失敗時の代替値フォールバックを提供する
  - 成功 Result から保持している成功値を取得できる
  - 失敗 Result では指定した代替値が例外なく返る
  - _Requirements: 2.1, 2.3, 2.4_
  - _Depends: 2.1_

- [ ] 3.2 失敗 Result からの直接取得成功値取得を拒否する
  - 失敗状態での成功値取得は専用例外により明確に失敗する
  - 失敗理由は例外メッセージに依存せず専用の参照手段で取得できる
  - _Requirements: 2.2_
  - _Depends: 1.3, 2.1, 3.1_

## 4. 変換・合成とログ結合

- [ ] 4.1 成功時のみ値変換し、失敗時は理由とログを透過する
  - 成功 Result の変換後も成功状態で新しい値が得られる
  - 失敗 Result は変換コールバックを実行せず、元の失敗理由と結合済みログを保持する
  - 変換前後のログが左から右へ順序を保って結合される
  - _Requirements: 3.1, 3.2, 4.4_
  - _Depends: 2.2_

- [ ] 4.2 Result を返す合成で短絡評価とログ結合を行う
  - 成功時のみ次の Result を評価し、その Result を返す
  - 失敗時は合成コールバックを呼ばず元の失敗理由と結合済みログを保持する
  - 評価済みログが左から右へ順序を保って結合される
  - _Requirements: 3.3, 3.4, 4.4_
  - _Depends: 4.1_

- [ ] 4.3 パイプ演算子向けの単引数 callable 形式を提供する
  - 1 引数で Result を受け取る callable として変換・合成を表現できる
  - 1.2 で確定した方針に従い、実行テストまたは文書サンプルで利用可能性を検証できる
  - _Requirements: 3.5_
  - _Depends: 1.2, 4.1, 4.2_

## 5. doo による逐次評価

- [ ] 5.1 Generator ベースの逐次評価と早期中断を実装する
  - すべて成功時は最終的な成功 Result が返る
  - 途中で失敗が現れたら以降を評価せず失敗 Result を返す
  - 成功完了時・失敗中断時とも、評価済み Result のログが順序を保って結合される
  - Result 以外の yield は不正利用として明確に拒否される
  - _Requirements: 5.1, 5.2, 5.3, 5.4, 5.5_
  - _Depends: 4.1, 4.2_

## 6. 静的解析向け公開契約

- [ ] 6.1 総称型注釈で成功値・失敗理由・ログ要素型を公開契約として表現する
  - 生成・変換・合成・逐次評価の戻り型が PHPDoc で追跡可能である
  - ログ collection の要素が LogEntry 契約に従うことが注釈で示される
  - 公開 API 全体で解析ツール向けの型情報が一貫している
  - _Requirements: 6.1, 6.2, 6.3, 6.4, 6.5_
  - _Depends: 2.1, 4.1, 4.2, 5.1_

## 7. PHPUnit による単体検証

- [ ] 7.1 生成・状態判定・排他性を検証する
  - 成功・失敗生成直後の状態判定と、成功値と失敗理由が同時に有効でないことをテストで確認できる
  - _Requirements: 1.1, 1.2, 1.3, 1.4, 1.5_
  - _Depends: 2.1_

- [ ] 7.2 値取得・代替値・unwrap 拒否を検証する
  - 成功時の値返却、失敗時の代替値、失敗時 unwrap の例外がテストで確認できる
  - _Requirements: 2.1, 2.2, 2.3, 2.4_
  - _Depends: 3.1, 3.2_

- [ ] 7.3 値変換・Result 合成の短絡とログ結合を検証する
  - 成功時の変換・合成、失敗時のコールバック未実行、ログの順序付き結合がテストで確認できる
  - _Requirements: 3.1, 3.2, 3.3, 3.4, 4.1, 4.2, 4.3, 4.4_
  - _Depends: 4.1, 4.2, 4.3_

- [ ] 7.4 doo の成功・失敗中断・ログ結合・不正 yield を検証する
  - 全成功時の最終 Result、途中失敗時の中断、ログ結合、非 Result yield の拒否がテストで確認できる
  - _Requirements: 5.1, 5.2, 5.3, 5.4, 5.5_
  - _Depends: 5.1_

- [ ] 7.5 失敗理由参照とログ collection の不変性を検証する
  - 失敗理由の取得と、返却したログ collection の変更が内部状態を変えないことがテストで確認できる
  - _Requirements: 4.1, 4.2, 4.3, 7.1_
  - _Depends: 2.2, 3.1_
